三、稳定性保障机制（6 大维度）

MeAgent 从 指令约束、错误熔断、流式容错、子Agent隔离、集成重试、状态管理 6 个维度保障输出稳定性。

1. Purpose 指令约束

预防 从源头减少模型「跑偏」

每个工具调用必须声明 purpose（意图），由 SYSTEM_INSTRUCTIONS 中的正反示例引导模型写出高质量的工具调用意图。

  • 13 个工具全部强制 purpose: str 参数
  • 正例："查询当前用户有权限的审批列表，用于回答用户的待办问题"
  • 反例："调用 dataitem 查询"（太笼统）

2. 错误熔断 & 自愈

容错 连续失败自动刹车

工具执行设有连续错误阈值（默认 16 次），超过后自动停止调用，避免死循环。

  • 单次失败：模型拿到错误详情，自动换一种方式重试
  • 连续失败：触发熔断，直接给用户友好提示
  • 所有错误分为 5 类，每类有对应的处理策略

3. 流式 Prefetch 容错

网络 三阶段错误保护

AG-UI 流式输出分预取 → 编码 → 流式三个阶段：

  • 预取阶段：先验证第一个事件，错误直接返回 HTTP 错误码
  • 编码阶段：防御性 fallback 到硬编码错误体
  • 流式阶段：中途出错追加 RUN_ERROR 事件

4. 子 Agent 运行时隔离

安全 互不干扰、独立容错

子 Agent 拥有独立的工具集、状态、和错误预算。

  • 每个子 Agent 独立 TodoState，无全局状态共享
  • 事件实时转发（思考/工具/结果），前端可追踪
  • 结构化结果提取（<result>），规范化输出

5. IntegHub 集成重试

集成 指数退避 + 超时保护

指数退避轮询（1s → 2s → 4s ... 最大 60s），总超时 600s + 连续失败 3 次熔断。

  • 敏感数据自动脱敏（API Key/Token/密码）
  • 异步任务支持：发起 → 轮询 → 获取结果
  • 日志截断 240 字符，防止信息泄露

6. 执行状态管理 & 取消

控制 全生命周期可控

全局状态追踪（pending → running → completed/failed），随时取消。

  • 同一会话同时只允许一个活跃执行
  • 取消原因分类：用户取消 / 超时 / 错误 / 断连
  • Token 用量实时统计（输入/输出/推理/缓存）

四、五层错误分类体系

所有错误被统一归类为 5 种类型，每种有 对应的处理策略和用户提示：

六、与流程化 Agent（如 Dify / Coze）的对比

维度MeAgent（自主式）流程化 Agent（Dify/Coze）
编排方式自主推理 + Skill 约束固定 DAG 流程图
灵活度高 — 同一请求可动态调整策略低 — 流程固定，改需求需改图
稳定性策略多维约束（Purpose + Skill + 熔断 + 隔离）流程固定 = 自带一致性
复杂任务天然支持（子 Agent 递归分派）需手动搭建复杂流程图
维护成本修改 Skill 文件即可需要专人维护流程图
模型依赖较高（需好模型才能自主推理）较低（流程固定，模型只填空）
适合场景企业级复杂业务（合同、审批、数据分析）简单固定流程（客服问答、表单填写）
核心差异：流程化方案的稳定性来自「流程固定」，但代价是灵活度低、维护成本高。MeAgent 的稳定性来自「多维约束 + 自愈机制」，在保持灵活度的同时实现可靠输出。

七、客户常见问题 & 回应话术

「你们怎么保证 Agent 输出的稳定性？」
我们从 6 个维度来保障：
  • 指令约束 — 每个工具调用必须声明意图（purpose），从源头引导模型行为
  • 技能系统 — 用 Skill 定义标准操作流程，确保同类任务输出一致
  • 错误熔断 — 连续失败自动停止，避免无意义重试；同时把错误信息给模型让它自动调整
  • 子 Agent 隔离 — 复杂任务拆成子任务，互不影响，一个子任务失败不会影响整体
  • 流式容错 — 三阶段错误保护，确保用户始终能看到有意义的反馈
  • 状态管理 — 全生命周期可追踪、可取消，Token 用量透明
「跟 Dify / Coze 这种流程化的方案比，你们的稳定性怎么样？」
这是两种不同的技术路线：
  • 流程化方案靠「流程固定」保证一致性 —— 好处是简单可预测，缺点是灵活度低
  • 我们靠「多维约束 + 自愈」保证一致性 —— Purpose 约束 + Skill 标准化 + 熔断保护
  • 在简单场景（如客服问答），两种方案稳定性相当
  • 在复杂场景（如合同审查、跨系统流程），流程化方案需要搭非常复杂的流程图；MeAgent 只需调整 Skill
  • 关键点：现在的大模型能力（Kimi K2.5 / DeepSeek V3.2 / Qwen3.5 / GLM-5 等）已经足够支撑自主推理的稳定性
「模型抽风了怎么办？比如一直调用错误的工具」
我们有自动熔断机制：
  • 连续错误超过阈值（默认 16 次）会自动停止，不会无限循环
  • 每次工具调用失败，详细的错误信息会反馈给模型，让它自己调整策略
  • 错误被分为 5 类，每类有不同的处理策略
  • 用户始终看到友好的中文提示，不会看到技术报错
「Agent 的输出格式能保证一致吗？」
通过 Skill 技能系统保证：
  • 每个业务场景定义一个 Skill，包含完整的指令、输出模板、正反例
  • Agent 每次执行前会加载对应 Skill，相当于「带着操作手册做事」
  • Skill 可以随时更新，不需要改代码，运营人员直接维护
「三方接口不稳定会影响 Agent 吗？」
不会，我们有专门的集成容错：
  • IntegHub 自带指数退避重试（1s → 2s → 4s ...最大 60s）
  • 连续失败 3 次自动熔断，总超时 600 秒
  • 所有敏感数据自动脱敏

八、一句话总结

MeAgent 的输出稳定性 = Purpose 指令约束（预防）+ Skill 标准化（一致）+ 错误熔断（兜底）+ 子 Agent 隔离（隔离）+ 流式容错（体验）+ 状态管理（可控）。

不是靠「固定流程」限制 Agent，而是靠「多维约束 + 自愈机制」让 Agent 在灵活应对复杂任务的同时，保持稳定、可靠、可控。
